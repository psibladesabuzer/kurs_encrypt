# План защиты курсового проекта на 5-7 минут

## 1. Введение, 40-60 секунд

Тема проекта: разработка браузерного расширения для локального шифрования почтовых вложений перед отправкой по электронной почте.

Проблема: при отправке вложений через веб-почту файл попадает во внешнюю систему в открытом виде. Если предварительно зашифровать его локально, то через почтовый сервис передается уже защищенный файл.

## 2. Демонстрация расширения, 1.5-2 минуты

Показать папку `browser_extension`.

Ключевые файлы:

- `manifest.json` - описание расширения для браузера;
- `popup.html` - форма выбора файла и ввода пароля;
- `popup.css` - оформление окна;
- `popup.js` - логика шифрования и расшифрования.

Что показать в браузере:

1. Открыть расширение.
2. Выбрать обычный файл.
3. Ввести пароль.
4. Получить файл с расширением `.aes256`.
5. Переключиться на расшифрование.
6. Выбрать `.aes256` и восстановить исходный файл.

Основной тезис: файл обрабатывается локально, без отправки на сервер.

## 3. Объяснение криптографии, 2-2.5 минуты

Открыть `kurs_crypto/crypto_service.py`.

Показать функции:

- `_derive_key()` - из пароля формируется 256-битный ключ через PBKDF2-HMAC-SHA256;
- `encrypt_file()` - файл шифруется AES-256-GCM;
- `decrypt_file()` - файл расшифровывается, а тег GCM проверяет пароль и целостность;
- `_build_header()` - создается заголовок с параметрами шифрования;
- `_read_file_prefix()` - заголовок читается при расшифровании.

Сказать по шагам:

1. Пользователь вводит пароль.
2. Генерируется случайная соль.
3. PBKDF2 многократно обрабатывает пароль с солью и получает ключ.
4. Генерируется случайный nonce для AES-GCM.
5. Файл шифруется.
6. В конец записывается тег аутентификации.
7. При неверном пароле или изменении файла проверка тега завершается ошибкой.

## 4. Формат файла, 40-60 секунд

Показать схему:

```text
MAGIC | VERSION | HEADER_LENGTH | HEADER_JSON | CIPHERTEXT | GCM_TAG
```

Пояснить:

- `MAGIC` и `VERSION` нужны, чтобы распознать формат;
- `HEADER_JSON` содержит salt, nonce, алгоритм, KDF и исходное имя файла;
- `CIPHERTEXT` - зашифрованные данные;
- `GCM_TAG` - контроль целостности.

## 5. Тестирование, 30-40 секунд

Открыть `tests/test_crypto_service.py`.

Сказать, что проверены:

- шифрование и последующее расшифрование;
- невозможность расшифровать неверным паролем;
- обнаружение поврежденного зашифрованного файла;
- отклонение файла неправильного формата.

Команда:

```powershell
py -3.9 -m unittest discover -s tests
```

## 6. Заключение, 30 секунд

Итог: разработан прототип расширения, которое позволяет локально шифровать почтовые вложения перед отправкой. Для защиты используются AES-256-GCM и PBKDF2-HMAC-SHA256. Реализация обеспечивает конфиденциальность файла и обнаружение изменения данных.

## Короткие ответы на возможные вопросы

Почему AES-GCM: он одновременно шифрует данные и проверяет их целостность.

Почему PBKDF2: пароль пользователя нельзя напрямую использовать как ключ AES; PBKDF2 получает из пароля криптографический ключ и усложняет перебор.

Зачем salt: чтобы одинаковые пароли давали разные ключи для разных файлов.

Зачем nonce: он нужен режиму GCM и должен быть уникальным для каждого шифрования.

Что передавать получателю: файл `.aes256` по почте, пароль - отдельным защищенным каналом.
